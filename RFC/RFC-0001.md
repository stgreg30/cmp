RFC-0001

Title

CMP: A Canonical Memory Protocol for Software Knowledge

Status

Draft

Authors

* Ashmi

Version

0.1

⸻

Abstract

Software repositories preserve source code.

Documentation preserves explanations.

Version control preserves history.

Issue trackers preserve discussion.

Observability platforms preserve execution.

Yet none preserve software knowledge itself.

Every year millions of developers independently rediscover the same architectural decisions, concurrency strategies, security boundaries, error handling techniques, and abstractions because today’s tools index syntax rather than meaning.

CMP proposes a canonical representation for software knowledge.

Rather than storing files, CMP stores semantic structures that remain invariant across programming languages and implementations.

The long-term objective is to build a global memory layer capable of discovering the underlying laws governing software systems.

⸻

Motivation

Git answers:

“What changed?”

GitHub answers:

“Who changed it?”

Documentation answers:

“How does it work?”

CMP attempts to answer an entirely different question.

“What knowledge does this software contain?”

Two authentication systems written in Rust and Go should be recognized as the same idea.

Two scheduling systems built twenty years apart should be comparable.

Two distributed databases should expose common invariants despite completely different implementations.

Current tooling cannot answer these questions because it indexes text.

CMP indexes meaning.

⸻

Problem Statement

Current software ecosystems organize information using syntax.
Repository
    ↓
Files
    ↓
Lines
    ↓
Tokens
This organization is excellent for editing software but poor for understanding it.

The same algorithm appears thousands of times under different names.

The same architectural decisions appear across unrelated ecosystems.

The same mistakes repeat because there is no shared semantic memory.

CMP introduces a representation independent of language.

⸻

Vision

CMP seeks to become for software knowledge what Git became for source code.

Git versions text.

CMP versions understanding.

Eventually an engineer—or an AI—should be able to ask:

“Show every independently discovered implementation of optimistic concurrency.”

or

“What architectural structures consistently survive systems larger than one million lines of code?”

or

“What software laws have the strongest empirical support?”

⸻

Principles

CMP is founded on eight principles.

1. Language Independence

Knowledge should survive translation between programming languages.

Meaning is primary.

Syntax is secondary.

⸻

2. Evidence First

Every claim stored by CMP must have evidence.

Evidence may include

* source repositories
* execution traces
* benchmarks
* tests
* documentation
* production incidents
* academic literature

⸻

3. No Handwritten Ontology

CMP should discover concepts rather than requiring humans to define every category beforehand.

The ontology evolves from evidence.

⸻

4. Provenance

Every observation remembers where it came from.

Nothing exists without provenance.

⸻

5. Falsifiability

Every proposed software law must be testable.

If evidence contradicts a law, the law changes.

⸻

6. Incremental Learning

Knowledge accumulates continuously.

Every repository processed should improve the global model.

⸻

7. Coordinate Freedom

Knowledge must remain stable under

* language changes
* framework changes
* operating system changes
* compiler changes

⸻

8. Human + AI

CMP is designed for both humans and intelligent systems.

Neither is considered the primary user.

⸻

Core Research Questions

CMP exists to investigate questions such as:

* What is the smallest semantic unit of software?
* What structures emerge independently?
* What architectural decompositions maximize longevity?
* Which error handling strategies minimize production failures?
* Which concurrency models remain stable across ecosystems?
* Can software laws be empirically discovered?
* Can software knowledge become cumulative?

⸻

Long-Term Goal

CMP aims to become a continuously evolving scientific knowledge base describing software systems in the same way biology describes living organisms or physics describes matter.

Rather than collecting code, CMP seeks to discover the invariant structures from which software is constructed.

⸻

Non-Goals

CMP is not

* another Git hosting service
* another documentation generator
* another vector database
* another code search engine
* another static analyzer

Those systems operate on software.

CMP attempts to understand software.

⸻

Closing Statement

Software engineering has accumulated billions of lines of code but almost no cumulative scientific understanding of why successful systems converge toward similar structures.

CMP proposes that software possesses discoverable invariant laws and that these laws can be extracted through large-scale semantic observation.

This document defines the first step toward building that scientific foundation.
